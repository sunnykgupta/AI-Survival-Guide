# Contributing to the AI Survival Guide

Thanks for wanting to make this guide better. The whole site is generated from Markdown files, so you do not need to touch any HTML or CSS to contribute. Edit a role file, open a pull request, and the site rebuilds itself when it merges.

## How the site is built

Every page on the site is generated by `build.py`. It reads:

- `content/roles/*.md` ... one Markdown file per role (this is where most contributions go)
- `content/sections/*.html` ... the static intro, model, and manifesto sections
- `template.html` ... the page shell, styles, and scripts
- `assets/` ... illustrations

It writes the finished site to `dist/`. You never edit `dist/` by hand. GitHub Actions runs the build for you on every push to `main`.

## The fastest way to contribute

1. Find the role you want to improve in `content/roles/`. Files are numbered by display order, for example `01-frontend.md`, `02-backend.md`.
2. Edit the relevant section. Keep the existing structure (see below).
3. Open a pull request describing what you changed and why.

That is it. Once a maintainer merges your change, the live site at https://ai-survival-guide.forbunnies.com updates automatically.

## Anatomy of a role file

Each role file has a small block of front matter at the top, then a series of `##` sections. Here is the shape:

```markdown
---
id: frontend
order: 1
title: Frontend Engineer
short: Frontend
icon: frontend
stack: React, TypeScript, CSS
---

## Lead
One punchy line that sets up the role.

## Intro
A paragraph or two of context. Plain text. Blank line between paragraphs.

## Layer 1 - Commodity: What AI now does for you
What an AI can do for this role today, with concrete examples.

## Layer 2 - Leverage: How to work with AI effectively
Where you add value by directing the tools.

## Layer 3 - Irreplaceable: What no AI can own
The judgment, taste, and context that does not automate away.

## Pull Quote
A single memorable sentence pulled out as a highlight.

## Real Talk
An honest, example-driven paragraph. No hype.

## Roadmap
1. **Days 1–30: Title.** What to do in the first month.
2. **Days 31–60: Title.** What to do next.
3. **Days 61–90: Title.** What to do to break out.

## Trap
The most common mistake people in this role are making right now.

## Bonus
One extra tip worth knowing.
```

### Front matter fields

| Field   | Meaning                                                           |
| ------- | ---------------------------------------------------------------- |
| `id`    | Unique slug, used for the anchor link (`#frontend`)              |
| `order` | Number that controls where the role appears (1 is first)         |
| `title` | Full role name shown in the header                               |
| `short` | Short label used in the navigation drawer                        |
| `icon`  | Icon key (see `role_icons.py` for available keys)                |
| `stack` | Comma-separated tools shown as chips                             |

## Adding a brand new role

1. Copy an existing file in `content/roles/` and rename it with the next number, for example `10-security.md`.
2. Update the front matter. Give it a unique `id` and the next `order` number.
3. Pick an `icon`. If you want a new icon, add an inline SVG entry to `role_icons.py` keyed by your new icon name. If you skip this, a default icon is used.
4. Fill in every `##` section.
5. Run the build locally to check it (see below), then open a pull request.

## House style

To keep the voice consistent across contributions:

- No em dashes. Use commas, periods, or "..." for a pause.
- Avoid the "this is not X, this is Y" sentence pattern.
- Stay factual and example-driven. Skip the hype and the filler.
- Use en dashes only for numeric ranges, for example "Days 1-30".

## Build it locally (optional)

You do not need anything except Python 3. There are no dependencies to install.

```bash
python3 build.py
```

This writes the full site to `dist/`. Open `dist/index.html` in your browser to preview your changes.

## What happens after you open a pull request

- A maintainer reviews the change.
- On merge to `main`, GitHub Actions runs `build.py` and deploys `dist/` to GitHub Pages.
- The live site updates within a couple of minutes.

You do not need to commit anything in `dist/`. It is generated in CI and is intentionally not tracked in git.
